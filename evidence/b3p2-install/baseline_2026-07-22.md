## The finding — **RETRACTED 2026-07-23. It was a capture defect, not a fact about the host.**

What this file originally recorded:

```
SeBatchLogonRight = <ABSENT FROM POLICY>     (secedit /export exit 0, principal_count = 0)
```

**That is wrong.** The true state, then and now:

```
SeBatchLogonRight = *S-1-5-32-544,*S-1-5-32-551,*S-1-5-32-559
                    Administrators, Backup Operators, Performance Log Users  (the Windows defaults)
```

### How the capture lied, and how that was proved

The baseline script exported the policy to a path from `[IO.Path]::GetTempFileName()`. That call **creates**
the file. `secedit /export` then writes into a path that already exists — and when it does, it emits UTF-16
**with no byte-order mark**. `Get-Content` has no BOM to detect, falls back to ANSI, and every line comes
back NUL-interleaved, so `-match "^SeBatchLogonRight"` matched nothing. It would have matched nothing for
*any* right; the script only asked about one, so the failure looked like a finding about that one.

Reproduced on the host on 2026-07-23, both branches in a single script against the same live policy:

```
--- EXACT BASELINE FORM: GetTempFileName() pre-creates the file ---
pre_existing_bytes=0   export_exit=0   post_export_bytes=3878
RESULT=SeBatchLogonRight = <ABSENT FROM POLICY>
principal_count=0
--- CONTROL: same everything, but the file does NOT pre-exist ---
export_exit=0          post_export_bytes=3928
RESULT=SeBatchLogonRight = *S-1-5-32-544,*S-1-5-32-551,*S-1-5-32-559
```

First bytes of each export, which is the whole mechanism:

```
precreated_first6 = 5B 00 50 00 72 00        "[P..." - NO BOM
fresh_first6      = FF FE 5B 00 55 00        BOM + "[U..."
precreated_asUTF16_hasRight = True           the line was in the bytes all along
```

### Proof that the host policy did **not** change between the two captures

| Check | Result |
|---|---|
| `secedit.sdb` last write | `2026-03-19T13:47:17Z` — four months *before* the baseline; no `/configure` has run |
| `C:\Windows\security\logs\scesrv.log` | **absent** — no policy-configuration operation has ever been logged |
| Principals now assigned | exactly the three Windows defaults; **no beta SID, no unexpected principal** |
| Beta identities / tasks / service | still `0` / `0` / ABSENT — nothing has been created |

Audit subcategory *Authorization Policy Change* is `No Auditing` on this host, so 4704/4705 events would not
exist either way; the `secedit.sdb` timestamp, not the event log, is what carries this claim.

**Consequence for the Phase 2 decision: none — the decision stands and is better supported.** The stated
premise ("the right is absent, so secedit would create a policy line from nothing") was wrong. The real
premise is worse for `secedit` and better for the LSA API: the line exists and holds three default
principals, so `secedit /configure` would have to reconstruct all three from a template *exactly* or
silently revoke batch logon machine-wide from whoever held it. `LsaAddAccountRights` adds one right to one
account, so it never has to know, infer or recreate the defaults — which is precisely what the Phase 2
stop-condition decision instructed.

**Consequence for evidence practice.** This is the same PowerShell 5.1 encoding trap that made all four
install scripts fail to parse on the host, in its other direction: there, BOM-less UTF-8 was read as ANSI;
here, BOM-less UTF-16 was. A verification step whose *negative* result is the finding must be shown to be
capable of producing a positive one. This one never was.
